# heist

A secure, encrypted secrets manager for the terminal — written in Rust.

heist stores secrets in a single encrypted file protected by a master password.
Every byte at rest is ciphertext. Nothing is ever written in plaintext.

```
$ heist set aws/access-key
Enter secret value: ************
Master password: ************
✓ Stored secret 'aws/access-key'

$ heist get aws/access-key
Master password: ************
AKIAIOSFODNN7EXAMPLE
```

## Features

- **AES-256-GCM** authenticated encryption — integrity is verified on every open
- **Argon2id** key derivation with OWASP-recommended parameters (64 MiB, 3 passes, 4 threads)
- **Encrypted audit log** — every read, write, copy, and exec is recorded inside the vault
- **Hierarchical keys** — namespace secrets with `/` separators (`aws/prod/access-key`)
- **Tag-based filtering** — tag secrets and filter `list`, `export`, and `exec` by tag
- **Clipboard integration** — copy to clipboard with automatic clearing after a configurable timeout
- **Secret injection** — run any subprocess with secrets injected as environment variables (`exec`)
- **Import / export** — `.env`, JSON, and YAML formats in both directions
- **Password rotation** — re-encrypt the entire vault with a new master password in one command
- **Atomic writes** — vault file is never partially written; a crash mid-save cannot corrupt it
- **Shell completions** — bash, zsh, fish, elvish, and PowerShell
- **CI-friendly** — password can be supplied via environment variable or file instead of a TTY prompt
- **Zero dependencies at runtime** — single statically-linked binary, no config daemon, no agent

## Security model

### Encryption

Each vault file contains:

| Field              | Notes                                           |
| ------------------ | ----------------------------------------------- |
| `version`          | File format version (currently `1`)             |
| `created_at`       | Vault creation timestamp (unencrypted)          |
| `salt`             | 256-bit Argon2id salt, generated once at `init` |
| `nonce`            | 96-bit AES-GCM nonce, regenerated on every save |
| `ciphertext`       | AES-256-GCM ciphertext of all secrets           |
| `audit_nonce`      | Separate nonce for the audit log blob           |
| `audit_ciphertext` | AES-256-GCM ciphertext of the audit log         |

Only the salt and timestamps are stored in the clear. Everything else — key names, values, descriptions, tags, and audit entries — is encrypted.

### Key derivation

```
master_password + salt → Argon2id → 256-bit AES key
```

Argon2id parameters:

| Parameter     | Value               | Why                                                |
| ------------- | ------------------- | -------------------------------------------------- |
| Memory        | 65 536 KiB (64 MiB) | OWASP minimum; raises cost for GPU/ASIC attacks    |
| Iterations    | 3                   | Multiplies time cost on top of memory              |
| Parallelism   | 4                   | Exploits multi-core while keeping serial cost high |
| Output length | 32 bytes            | Full AES-256 key                                   |

### Nonce handling

A fresh 96-bit random nonce is generated by `OsRng` on every vault write. Using a new nonce per write means that even if two saves produce identical plaintext, the ciphertexts differ. The nonce is not secret and is stored alongside the ciphertext.

### Atomic writes

Saves are performed via **write-to-tmp then rename**:

```
vault.heist.tmp  ←  new encrypted content
rename → vault.heist
```

A process crash or power loss between the write and the rename leaves the original vault intact. The `.tmp` file is discarded on the next open if present.

### Key zeroization

The 256-bit AES key is held in a `Zeroizing<[u8; 32]>` wrapper from the [`zeroize`](https://docs.rs/zeroize) crate, which overwrites the memory with zeros when the value is dropped. Secret values in the `Secret` struct are also `Zeroize + ZeroizeOnDrop`.

### What is NOT protected

- **Memory inspection** by a process running as the same user (e.g. `ptrace`, `/proc/mem`).
  heist provides no protection against an attacker who already has code execution as you.
- **Clipboard sniffing** by other processes. The auto-clear on `copy`/`get --clip` is a
  convenience measure, not a security guarantee.
- **Swap / core dumps**. Consider enabling full-disk encryption and disabling swap if you
  operate in a high-threat environment.

## Installation

No Rust required. Pick the one-liner for your platform.

### macOS and Linux

```sh
curl -fsSL https://raw.githubusercontent.com/iamdainwi/heist/master/install.sh | sh
```

The script auto-detects your OS and CPU architecture, downloads the right
pre-built binary, verifies its SHA-256 checksum, and installs to
`/usr/local/bin` (or `~/.local/bin` if you don't have write access there).

**Install a specific version:**

```sh
HEIST_VERSION=v0.2.0 curl -fsSL https://raw.githubusercontent.com/iamdainwi/heist/master/install.sh | sh
```

**Install to a custom directory:**

```sh
HEIST_INSTALL_DIR=~/.local/bin curl -fsSL https://raw.githubusercontent.com/iamdainwi/heist/master/install.sh | sh
```

### Windows (PowerShell)

```powershell
irm https://raw.githubusercontent.com/iamdainwi/heist/master/install.ps1 | iex
```

Downloads the Windows binary, verifies its checksum, installs to
`%LOCALAPPDATA%\heist\bin`, and adds that directory to your user `PATH`
permanently.

### Homebrew (macOS)

```sh
brew install iamdainwi/tap/heist
```

### Cargo (requires Rust)

```sh
cargo install heist
```

### Build from source

See [Building from source](#building-from-source).

## Quick start

```bash
# 1. Create a vault (stored at ~/.heist/vault.heist by default)
heist init

# 2. Store your first secret
heist set github/token --description "GitHub personal access token" --tags "github,prod"
# Prompts for the secret value (hidden) and your master password

# 3. Retrieve it
heist get github/token

# 4. Copy to clipboard (auto-clears after 45 seconds)
heist copy github/token

# 5. List all secrets
heist list

# 6. Store more, organised by namespace
heist set aws/prod/access-key
heist set aws/prod/secret-key
heist set database/prod/url

# 7. List just the AWS secrets
heist list aws/

# 8. Inject into a subprocess
heist exec aws/prod/access-key aws/prod/secret-key -- terraform apply
```

## Commands

### init

Create a new vault. Prompts for a master password (with confirmation).

```
heist init [--force]
```

| Flag          | Description                                   |
| ------------- | --------------------------------------------- |
| `-f, --force` | Overwrite an existing vault without prompting |

```bash
heist init
heist init --force          # overwrite, e.g. for a fresh start in CI
heist init --vault /tmp/test.heist
```

---

### set

Store or update a secret. If the key already exists, its value is updated and `updated_at` is bumped; `created_at` is preserved.

```
heist set <KEY> [--value <VALUE>] [--description <TEXT>] [--tags <TAG,...>]
```

| Flag                         | Description                                                                    |
| ---------------------------- | ------------------------------------------------------------------------------ |
| `-v, --value <VALUE>`        | Secret value as a flag (avoid for sensitive values — appears in shell history) |
| `-d, --description <TEXT>`   | Human-readable description                                                     |
| `-t, --tags <TAG1,TAG2,...>` | Comma-separated tags                                                           |

**Value resolution order:**

1. `--value` flag
2. Stdin pipe (preferred — does not appear in shell history)
3. Interactive hidden prompt

```bash
# Interactive prompt (safest)
heist set DATABASE_URL

# Pipe from a command
aws ssm get-parameter --name /prod/db-url --query 'Parameter.Value' --output text \
  | heist set database/prod/url

# From a file
heist set tls/cert < /etc/ssl/certs/server.crt

# Inline (appears in shell history — avoid for secrets)
heist set APP_ENV --value production

# With metadata
heist set stripe/secret-key \
  --description "Stripe live secret key" \
  --tags "billing,prod,stripe"
```

### get

Retrieve a secret value. Prints to stdout by default, suitable for piping.

```
heist get <KEY> [--clip] [--timeout <SECS>] [--meta]
```

| Flag               | Description                                                             |
| ------------------ | ----------------------------------------------------------------------- |
| `-c, --clip`       | Copy to clipboard instead of printing                                   |
| `--timeout <SECS>` | Seconds before clipboard is cleared (default: `45`, only with `--clip`) |
| `-m, --meta`       | Print description, tags, and timestamps alongside the value             |

```bash
# Print to stdout
heist get github/token

# Pipe into another command
heist get database/prod/url | psql

# Copy to clipboard, clear after 60 seconds
heist get stripe/secret-key --clip --timeout 60

# Show full metadata
heist get aws/prod/access-key --meta

# Use in a script
export DATABASE_URL=$(heist get database/prod/url)
```

### list

List secrets in a formatted table.

```
heist list [PREFIX] [--tag <TAG>]... [--json]
```

| Flag              | Description                                         |
| ----------------- | --------------------------------------------------- |
| `PREFIX`          | Only show secrets whose key starts with this string |
| `-t, --tag <TAG>` | Filter by tag (repeatable; all tags must match)     |
| `--json`          | Output as a JSON object (key → metadata)            |

```bash
heist list                        # all secrets
heist list aws/                   # only AWS secrets
heist list --tag prod             # only production secrets
heist list --tag prod --tag db    # production AND database secrets
heist list aws/ --tag prod        # AWS + production
heist list --json                 # machine-readable output
```

**Example output:**

```
┌──────────────────────────────────────────────────────────────────────┐
│ KEY                    DESCRIPTION          TAGS            UPDATED  │
╞══════════════════════════════════════════════════════════════════════╡
│ aws/prod/access-key    AWS IAM access key   aws, prod       2024-... │
│ aws/prod/secret-key    AWS IAM secret key   aws, prod       2024-... │
│ database/prod/url      Production DB URL    db, prod        2024-... │
│ github/token           GitHub PAT           github, prod    2024-... │
└──────────────────────────────────────────────────────────────────────┘
  4 secrets
```

### remove

Delete a secret permanently. Prompts for confirmation unless `--yes` is passed.

```
heist remove <KEY> [--yes]
```

| Flag        | Description                  |
| ----------- | ---------------------------- |
| `-y, --yes` | Skip the confirmation prompt |

```bash
heist remove old/api-key
heist rm old/api-key           # alias
heist remove old/api-key --yes # no prompt
```

### copy

Copy a secret value to the clipboard. A background thread clears the clipboard after the timeout elapses, but only if the clipboard still holds the copied value (i.e. the user hasn't copied something else in the meantime).

```
heist copy <KEY> [--timeout <SECS>]
```

| Flag               | Description                                              |
| ------------------ | -------------------------------------------------------- |
| `--timeout <SECS>` | Seconds before clipboard is auto-cleared (default: `45`) |

```bash
heist copy github/token
heist cp   github/token                 # alias
heist copy stripe/secret-key --timeout 15
```

---

### exec

Execute a command with one or more secrets injected as environment variables. Key names are transformed to uppercase env-var format:

```
aws/access-key   →   AWS_ACCESS_KEY
prod/db-url      →   PROD_DB_URL
DATABASE.URL     →   DATABASE_URL
```

The transformation rules: uppercase everything; replace `/`, `-`, and `.` with `_`.

```
heist exec <KEY>... -- <COMMAND> [ARGS...]
```

Secrets are injected **only** into the child process environment. They are not exported to the parent shell and do not appear in shell history.

```bash
# Terraform with AWS credentials
heist exec aws/prod/access-key aws/prod/secret-key -- terraform apply

# Rails with database URL
heist exec database/prod/url -- rails db:migrate

# Docker build with build args
heist exec registry/token -- \
  docker build --secret id=token,env=REGISTRY_TOKEN .

# Any script
heist exec github/token -- ./deploy.sh
```

### import

Import secrets from a `.env`, JSON, or YAML file. The format is detected from the file extension when `--format` is omitted.

```
heist import <FILE> [--format env|json|yaml] [--namespace <NS>] [--overwrite]
```

| Flag                   | Description                                     |
| ---------------------- | ----------------------------------------------- |
| `-f, --format`         | Force a specific format (`env`, `json`, `yaml`) |
| `-n, --namespace <NS>` | Prepend a namespace prefix to all imported keys |
| `--overwrite`          | Overwrite existing secrets without prompting    |

**Supported input formats:**

`.env` — standard `KEY=VALUE` pairs, `#` comments, single/double-quoted values:

```env
# .env
DATABASE_URL="postgres://localhost/mydb"
SECRET_KEY=abc123
API_TOKEN='ghp_xyz'
```

JSON — flat object mapping key names to string values:

```json
{
  "DATABASE_URL": "postgres://localhost/mydb",
  "SECRET_KEY": "abc123"
}
```

YAML — flat mapping:

```yaml
DATABASE_URL: postgres://localhost/mydb
SECRET_KEY: abc123
```

```bash
heist import .env
heist import .env --namespace prod        # all keys become prod/KEY
heist import secrets.json --overwrite
heist import secrets.yaml --format yaml
```

### export

Export secrets to stdout or a file in `.env`, JSON, or YAML format. Filters by prefix or tag.

```
heist export [PREFIX] [--format env|json|yaml] [--output <FILE>] [--tag <TAG>]...
```

| Flag                  | Description                                           |
| --------------------- | ----------------------------------------------------- |
| `-f, --format`        | Output format (default: `env`)                        |
| `-o, --output <FILE>` | Write to a file instead of stdout                     |
| `PREFIX`              | Only export secrets whose key starts with this prefix |
| `-t, --tag <TAG>`     | Filter by tag (repeatable)                            |

```bash
# Print all secrets as .env
heist export

# Only AWS secrets, as JSON
heist export aws/ --format json

# All prod secrets to a file
heist export --tag prod --format env --output .env.prod

# Pipe into a docker run
heist export --format env | xargs -I{} docker run --env-file /dev/stdin myapp

# YAML for Kubernetes secret generation
heist export k8s/ --format yaml
```

> **Warning:** exported secrets are plaintext. Handle the output as you would any sensitive credential file. Prefer `exec` for subprocess injection instead of exporting to disk when possible.

### log

Display the encrypted audit log. Every operation that reads or modifies the vault is recorded. The log itself is encrypted with the same key as the secret data and is stored inside the vault file.

```
heist log [--limit <N>] [--key <KEY>]
```

| Flag              | Description                                                     |
| ----------------- | --------------------------------------------------------------- |
| `-l, --limit <N>` | Maximum number of entries to show (default: `50`, newest first) |
| `-k, --key <KEY>` | Filter entries by secret key prefix                             |

```bash
heist log
heist log --limit 100
heist log --key aws/            # only entries for AWS secrets
heist log --key github/token    # only entries for this exact key
```

**Example output:**

```
┌─────────────────────────────────────────────────────────┐
│ TIMESTAMP             ACTION   KEY               NOTE   │
╞═════════════════════════════════════════════════════════╡
│ 2024-11-01 14:23:01   EXEC     aws/prod/...,..   cmd=tf │
│ 2024-11-01 14:22:58   GET      aws/prod/acc...   —      │
│ 2024-11-01 09:11:44   SET      stripe/secret-..  —      │
│ 2024-11-01 09:10:02   INIT     *                 —      │
└─────────────────────────────────────────────────────────┘
  Showing 4 of 4 entries
```

Logged actions: `INIT`, `SET`, `GET`, `COPY`, `DELETE`, `EXEC`, `EXPORT`, `IMPORT`, `ROTATE`.

### rotate

Change the master password. Generates a new Argon2id salt and derives a new AES-256 key, then re-encrypts the entire vault (all secrets and the audit log) with the new key. The old password cannot decrypt the vault after rotation.

```
heist rotate
```

```bash
heist rotate
# Current master password: ****
# Enter master password:   ****   (new)
# Confirm master password: ****
# ✓ Master password rotated. Vault re-encrypted with new key.
```

> Rotate regularly and immediately if you suspect your master password has been compromised.

## Key format

Keys identify secrets. They follow these rules:

- **1 to 256 characters**
- **Segments** separated by `/` (used as a namespace separator)
- **Each segment** matches `[a-zA-Z0-9_.@-]+`
- **No leading or trailing `/`**
- **No consecutive `//`**

Valid examples:

```
TOKEN
DATABASE_URL
aws/access-key
prod/db/PASSWORD
github/token
stripe/prod/secret-key
service.account@prod
```

Invalid examples:

```
                     # empty
/leading-slash       # starts with /
trailing-slash/      # ends with /
double//slash        # consecutive slashes
has space            # spaces not allowed
has$dollar           # $ not allowed
```

### Key-to-env-var mapping (`exec`)

When injecting secrets as environment variables via `exec`, the key is transformed:

| Transformation | Example                             |
| -------------- | ----------------------------------- |
| Uppercase      | `aws/access-key` → `AWS/ACCESS-KEY` |
| `/` → `_`      | `AWS/ACCESS-KEY` → `AWS_ACCESS_KEY` |
| `-` → `_`      | `AWS_ACCESS-KEY` → `AWS_ACCESS_KEY` |
| `.` → `_`      | `db.url` → `DB_URL`                 |
| `@` → `_AT_`   | `svc@prod` → `SVC_AT_PROD`          |

## Vault location

The vault is a single encrypted JSON file. The default location is:

| Platform      | Default path                       |
| ------------- | ---------------------------------- |
| Linux / macOS | `~/.heist/vault.heist`             |
| Windows       | `%USERPROFILE%\.heist\vault.heist` |

Override it with the `--vault` flag or the `HEIST_VAULT` environment variable:

```bash
heist --vault /path/to/custom.heist list
HEIST_VAULT=/path/to/custom.heist heist list
```

`HEIST_VAULT` takes precedence over the default location. `--vault` takes precedence over everything.

## Multiple vaults

heist supports multiple independent vaults, each with its own master password. Use `--vault` or `HEIST_VAULT` to select which vault to operate on.

```bash
# Separate vaults for different environments
heist --vault ~/.heist/work.heist   set github/token
heist --vault ~/.heist/personal.heist set github/token

# Or with env vars in different shell sessions
export HEIST_VAULT=~/.heist/work.heist
heist set github/token

# Or use shell aliases
alias heist-work='heist --vault ~/.heist/work.heist'
alias heist-personal='heist --vault ~/.heist/personal.heist'
```

## CI and automation

In CI environments there is no controlling TTY for an interactive password prompt. heist supports two non-interactive password mechanisms:

### Option 1 — password file (recommended)

Store the password in a file with restricted permissions, then point heist at it:

```bash
# Set up once
echo "my-ci-password" > ~/.heist-ci-password
chmod 600 ~/.heist-ci-password

# Use in CI
export HEIST_PASSWORD_FILE=~/.heist-ci-password
heist init --force
heist set DEPLOY_TOKEN --value "$TOKEN"
heist exec DEPLOY_TOKEN -- ./deploy.sh
```

In GitHub Actions, store the password as a secret and write it to a file:

```yaml
- name: Write heist password file
  run: echo "${{ secrets.HEIST_PASSWORD }}" > /tmp/heist.pw

- name: Deploy
  env:
    HEIST_VAULT: .heist/ci.heist
    HEIST_PASSWORD_FILE: /tmp/heist.pw
  run: heist exec DEPLOY_TOKEN -- ./deploy.sh
```

### Option 2 — environment variable

```bash
export HEIST_MASTER_PASSWORD="my-ci-password"
heist list
```

> **Security note:** Environment variables are visible to all processes running as the same user on Linux (`/proc/<pid>/environ`) and may appear in CI logs if not masked. Prefer `HEIST_PASSWORD_FILE` for production CI pipelines.

### Example: GitLab CI

```yaml
deploy:
  script:
    - echo "$HEIST_PASSWORD" > /tmp/heist.pw
    - chmod 600 /tmp/heist.pw
    - export HEIST_PASSWORD_FILE=/tmp/heist.pw
    - heist exec aws/prod/access-key aws/prod/secret-key -- terraform apply
  after_script:
    - rm -f /tmp/heist.pw
```

## Quantum resistance

heist uses **only symmetric cryptography**. There is no public-key component (no RSA, no ECDH, no ECDSA). This matters because:

- **Shor's algorithm** — the quantum algorithm that breaks public-key cryptography — has nothing to attack here.
- **Grover's algorithm** — the only quantum speedup relevant to symmetric crypto — provides a quadratic speedup on brute-force key search. Against AES-256, this reduces effective security from 2²⁵⁶ to 2¹²⁸ operations. NIST's post-quantum guidance considers 128-bit quantum security acceptable for symmetric encryption.

| Primitive                 | Quantum threat     | Post-quantum strength       |
| ------------------------- | ------------------ | --------------------------- |
| AES-256-GCM               | Grover (quadratic) | ~128-bit quantum security ✓ |
| Argon2id (256-bit output) | Grover (quadratic) | ~128-bit quantum security ✓ |
| Public-key cryptography   | Shor (exponential) | **Not used** ✓              |

No algorithm changes are needed for post-quantum security. A sufficiently strong master password remains the primary attack surface.

## Threat model

### Protected against

- An attacker with **read access to the vault file** (e.g. stolen backup, shared filesystem).
  Without the master password, the file is opaque ciphertext. Brute-forcing Argon2id at the configured parameters is computationally expensive by design.

- **Tampering with the vault file**. AES-GCM is an authenticated cipher. Any modification to the ciphertext or nonce causes decryption to fail with an explicit error.

- **Accidentally leaving secrets in shell history**. The recommended `set` workflow reads the value from a hidden prompt or stdin pipe, neither of which appears in shell history.

- **Clipboard exposure**. `copy` and `get --clip` spawn a background thread that clears the clipboard after the timeout, and only if the clipboard still holds the copied value.

### Not protected against

- An attacker with **code execution as your user**. Memory inspection, `ptrace`, and `/proc/mem` access can extract the in-memory decrypted key and secrets. Full-disk encryption and OS-level process isolation are the appropriate controls here.

- **Weak master passwords**. Argon2id makes brute-force expensive but not impossible. Use a long, random passphrase. A password manager for your password manager is not ironic — it's prudent.

- **Malicious processes reading the clipboard**. The auto-clear is best-effort.

- **Swap / core dumps** containing in-memory secrets. Enable full-disk encryption if this is a concern.

## Running tests

```bash
# Unit tests + integration tests
cargo test

# Unit tests only (faster, no tempfile I/O)
cargo test --lib

# Integration tests only
cargo test --test integration_tests

# With output (useful for debugging)
cargo test -- --nocapture
```

The test suite covers:

- Encrypt / decrypt round-trips (correct key, wrong key, tampered ciphertext)
- Vault init, open, persist, and re-open
- Secret CRUD (create, update, delete, tag filtering)
- Password rotation (old password rejected, new password accepted, data preserved)
- Atomic save correctness across multiple consecutive writes
- Audit log persistence and 1 000-entry cap
- Key validation (valid and invalid patterns)
- Key-to-env-var name transformation
